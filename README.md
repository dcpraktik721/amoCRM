# Реализация amoCRM для клиники Практик. Комплект документов

Комплект артефактов для внедрения amoCRM в стоматологической клинике "Практик". Документы рассчитаны на руководителя клиники, Координатора лечения, маркетолога, администраторов, технического интегратора и подрядчика по настройке amoCRM.

## Что создано

| Файл | Назначение | Кому |
|---|---|---|
| `amo-crm-implementation-infographic.html` | Автономная интерактивная HTML-инфографика. Открывается локально в браузере. | Руководитель, Координатор лечения, Маркетолог, Администратор, подрядчик |
| `docs/amo-crm-diagrams.md` | Mermaid-диаграммы: архитектура, путь пациента, 5 воронок, потоки Calltouch -> Альбато -> amoCRM, DentalPRO -> amoCRM, роли, отчеты. | Технический интегратор, подрядчик |
| `docs/amo-crm-visual-tz.md` | Визуальное ТЗ для подрядчика: что реализовать, какие поля, роботы, блокировки, SLA, права. | Подрядчик, Технический интегратор |
| `docs/amo-crm-acceptance-checklist.md` | Чек-лист приемки. 20 тест-кейсов и сводная таблица. | Подрядчик, Координатор лечения, Руководитель |
| `README.md` | Этот файл. Точка входа. | Все роли |

## Как открыть HTML-инфографику

### Онлайн

- Timeweb App: https://dcpraktik721-amocrm-c926.twc1.net/
- Резервная публикация GitHub Pages: https://dcpraktik721.github.io/amoCRM/
- Репозиторий: https://github.com/dcpraktik721/amoCRM

Если Timeweb-домен не открывается сразу после создания приложения, подождать несколько минут до обновления DNS.

### Локально

1. Открыть файл `amo-crm-implementation-infographic.html` в любом современном браузере (Chrome, Safari, Firefox, Edge).
2. Файл автономный: внешние библиотеки и интернет не требуются.
3. Якорное меню вверху позволяет переходить между блоками. Кнопка "Наверх" доступна при прокрутке.
4. Для печати использовать стандартный диалог печати браузера. CSS @media print включает компактный режим.

## Какие документы смотреть и в каком порядке

1. Руководитель клиники - открывает HTML-инфографику и читает блоки `Архитектура`, `Воронки`, `Отчеты`, `Критерии приемки`.
2. Координатор лечения - HTML-инфографика, блоки `Путь пациента`, `Воронки`, `Автоматизации`, `Роли`. Затем `docs/amo-crm-acceptance-checklist.md`.
3. Маркетолог - HTML-инфографика, блок `Источники`. Затем `docs/amo-crm-visual-tz.md` разделы 4 и 11.
4. Администратор - HTML-инфографика, блоки `Воронки`, `Поля`, `Справочники причин`.
5. Технический интегратор и подрядчик - `docs/amo-crm-visual-tz.md` целиком, затем `docs/amo-crm-diagrams.md`, далее `docs/amo-crm-acceptance-checklist.md`.

## Как использовать материалы подрядчику

1. Прочитать `docs/amo-crm-visual-tz.md` целиком.
2. Сверить понимание архитектуры с диаграммами в `docs/amo-crm-diagrams.md`.
3. Реализовать в amoCRM пять воронок, поля, справочники, роботов и блокировки переходов.
4. Подключить интеграции Calltouch -> Альбато -> amoCRM и DentalPRO -> amoCRM.
5. Настроить обратную передачу статусов в Calltouch через Альбато.
6. Выдать доступы по ролям. Старший администратор не используется. Управленческие функции по удержанию выполняет Координатор лечения.
7. Пройти все 20 тест-кейсов из `docs/amo-crm-acceptance-checklist.md`.
8. Подписать акт по разделу `Условия подписания акта`.

## Что нельзя менять без согласования

- Состав и порядок воронок (5 воронок: Первичные обращения, План лечения, Лечение, Повторные визиты, Реактивация).
- Состав обязательных полей контакта и сделки.
- Закрытые справочники причин отказа, неявки и переноса.
- Блокировки переходов между этапами.
- Принцип границ систем: amoCRM не подменяет DentalPRO, DentalPRO не подменяет amoCRM.
- Принцип двух полей источника: `Источник первичного привлечения` (защищенное от перезаписи) и `Источник текущей сделки`.
- Замена `Старший администратор` на `Координатор лечения` во всех ролях и автоматизациях.

Любые изменения этих пунктов согласуются с руководителем проекта в письменном виде до внесения.

## Что в этом комплекте отсутствует и не должно появиться

- Реальные токены, ключи, пароли, закрытые URL.
- Вымышленные API DentalPRO и эндпоинты, не подтвержденные документацией.
- Англоязычный интерфейс. Допустимы только технические термины: amoCRM, DentalPRO, Calltouch, OAuth, API, webhook, UTM, ROMI, SLA.
- Длинные тире. Используется обычный дефис.

## Контроль соответствия

QA-аудитор перед передачей в работу подрядчику проверяет:
- наличие всех 5 воронок,
- все 11 групп источников,
- роли без `Старшего администратора`,
- архитектуру Calltouch -> Альбато -> amoCRM и обратный поток,
- DentalPRO как мастер-систему медицины и расписания,
- обязательные поля,
- справочники причин,
- блокировки переходов,
- критерии приемки,
- отсутствие токенов и закрытых URL,
- русский язык во всех текстах.
